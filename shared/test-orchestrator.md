# Test Orchestrator Shared Workflow

この文書は、AI エージェントがテスト計画、ユニットテスト、E2E テスト、目視確認、結果記録を一貫して扱うための共通本文である。特定ツール固有のサブエージェント、コマンド登録、インストール先は各ツールの入口ファイルに置く。

## 発動条件

- テスト計画、テスト実行、カバレッジ改善、E2E、動作確認、目視確認を依頼されたとき
- `test`, `coverage`, `unit test`, `E2E`, `テスト`, `カバレッジ`, `動作確認` などが含まれるとき

## 全体フロー

1. 対象機能と変更範囲を把握する
2. テスト計画を作成し、Unit/E2E に振り分ける
3. ユニットテストを実行し、失敗や不足を修正または報告する
4. 必要に応じて E2E テストを実行し、テスト用 DB/環境を復元する
5. 動画やスクリーンショットが必要な場合は目視確認素材を生成する
6. `test-results.md` に日付付きで結果を記録する

ツールがサブエージェントや別プロセスを持つ場合は各フェーズを分担してよい。持たない場合は同じエージェントが順番に実行する。

## テスト計画

対象コードを読み、主要な機能、依存関係、エッジケースを整理する。計画には以下を含める。

- 対象機能と対象ファイル
- 正常系、異常系、境界値、エッジケース
- Unit Test と E2E Test の振り分け
- 外部 API、DB、外部サービスのモック方針
- 優先度とカバレッジ見込み

Unit Test に適するもの:

- 単一関数、メソッド、ビジネスロジック
- 入力バリデーション、データ変換、計算処理
- モック可能な外部依存

E2E Test に適するもの:

- 複数コンポーネント連携
- ユーザーフロー全体
- 認証、認可、ファイル操作、WebSocket
- 実ブラウザや実 DB 操作が意味を持つ検証

テンプレートが必要な場合は `shared/test-plan-template.md` を使う。

## ユニットテスト

原則としてコンテナ内で実行する。ホストでの直接実行が避けられない場合は、理由と影響を報告してから進める。

検出の優先順位:

- Frontend: `vitest.config.*`、`jest.config.*`、`package.json` の `test` script
- Backend: `pytest.ini`、`pyproject.toml`、`setup.cfg`、`conftest.py`

標準目標:

- Unit coverage: 90%
- 90% 未満の場合は、未カバー箇所と追加テスト案を出してユーザーに相談する
- 外部 API と DB はモックする
- 本番 DB と本番外部サービスには接続しない

実装変更を行った場合は、変更ファイル、理由、影響範囲を必ず報告する。

## E2E テスト

E2E はユニットテストでカバーできないフローに絞る。テスト用 DB または隔離されたテスト環境を使い、本番 DB には接続しない。

必須事項:

- 現在の DB/環境設定を記録する
- テスト用設定へ切り替える
- コンテナやサービスを再起動して切り替えを確認する
- テスト後に元の設定へ復元する
- 復元確認をログまたはレポートに残す

本番 DB、復元失敗、テスト用 DB 作成失敗、コンテナ起動失敗を検出した場合は、即座に停止して報告する。

## 目視確認

ユーザーが「動作確認」「動画」「目視」「スクショ」などを求めた場合に実行する。成果物は以下を揃える。

- シナリオごとの動画
- シナリオごとのスクリーンショット
- 動画とスクリーンショットを一覧できる HTML プレイヤー
- 確認チェックリスト

Playwright を使う場合は、動画を環境変数で切り替え可能にし、スクリーンショットはテスト出力ディレクトリに保存する。Claude 用の既存実装では `generate-player.ts` と `run-visual-verify.sh` を使う。

## 結果記録

テスト実行後は、プロジェクトルートの `test-results.md` に追記する。存在しない場合は作成する。

```markdown
## YYYY-MM-DD HH:MM (JST)

### Unit Tests
| Category | Passed | Failed | Skipped | Total |
|----------|--------|--------|---------|-------|
| Frontend | X | X | X | X |
| Backend  | X | X | X | X |

### E2E Tests
| Passed | Failed | Skipped | Total |
|--------|--------|---------|-------|
| X | X | X | X |

### Summary
- All tests passed / X failures detected
- Coverage: Frontend XX%, Backend XX%

### Fixes Applied
- `file.ts:line` - 修正内容

### Notes
- 特記事項

---
```

## 重要ルール

1. 本番 DB を使わない
2. 外部 API/DB はユニットテストではモックする
3. 失敗テストは原因を分類し、テスト問題か実装問題かを明確にする
4. 実装変更は必ず報告する
5. E2E 後は環境復元を確認する
6. カバレッジ 90% 未達は自己判断で流さず相談する
7. テスト結果は `test-results.md` に記録する
