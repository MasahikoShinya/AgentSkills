# Code Review Shared Workflow

この文書は、AI エージェントがコードレビューを行う際の共通観点、独立レビュー、クロスレスポンス、統合レポートの形式を定義する。Claude Code、Codex CLI、Gemini CLI などの入口ファイルは、この本文を読んだうえで各ツールの実行能力に合わせて運用する。

## 発動条件

- 「レビューして」「コードを見て」「PR をレビュー」「変更を確認して」などの依頼
- `review`, `code review`, `レビュー`, `コード確認`, `コードチェック` などが含まれるとき

## 基本方針

- バグ、セキュリティ、設計、保守性、パフォーマンス、エッジケースを優先する
- 好みやスタイルの指摘は、実害や保守上の根拠がある場合だけ扱う
- 指摘にはファイル、行番号、コード上の根拠、影響、修正案を含める
- 矛盾する見解は AI が裁定しない。根拠を並べ、人間の判断に委ねる
- 複数モデルや複数エージェントが使える場合は、独立レビューとクロスレスポンスを分ける
- 複数レビュアーが使えない場合は、単独レビューであることを明記し、クロスレスポンスを捏造しない

## 対象コードの特定

ユーザーの指示に応じて対象を判定する。

| 入力 | 対象 |
|---|---|
| ファイル指定 | 指定ファイル全体 |
| 変更レビュー | `git diff` と `git diff --staged` |
| PR 指定 | PR の差分 |
| 関数指定 | 指定シンボル周辺 |
| 指定なし | 未コミット変更 |

## レビュー観点

| 観点 | チェック内容 |
|---|---|
| バグ・ロジックエラー | 条件分岐漏れ、off-by-one、null/undefined、型不整合、状態遷移漏れ |
| セキュリティ | インジェクション、認証・認可漏れ、機密情報露出、安全でないデフォルト |
| 設計・保守性 | 責務分離、命名、複雑度、重複、局所修正で済まない設計負債 |
| パフォーマンス | N+1、不要な再計算、メモリリーク、過剰な同期処理 |
| エッジケース | 空配列、境界値、並行処理、リトライ、エラーパス |
| テスト | 重要分岐の未テスト、回帰検出不能、過度なモック |

## 独立レビュー

複数レビュアーが使える場合は、互いの結果を見せずにレビューさせる。単独の場合はこの形式でレビュー結果を出す。

```markdown
## Independent Review

### Findings

#### [Critical/Warning/Info] 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **観点**: バグ / セキュリティ / 設計 / パフォーマンス / エッジケース / テスト
- **内容**: 具体的な問題
- **根拠**: コードの具体的な箇所と、なぜ問題か
- **影響**: 発生しうる不具合や運用上のリスク
- **提案**: 修正案
```

重要度の目安:

- Critical: データ破壊、セキュリティ事故、主要機能停止、本番障害につながる
- Warning: 条件付きの不具合、保守性や運用に明確な悪影響がある
- Info: 低リスクだが直す価値がある

## クロスレスポンス

別レビュアーの指摘を検証できる場合だけ行う。

```markdown
## Cross-Response

### 相手の指摘: [指摘タイトル]
- **判定**: 同意 / 反論 / 補足
- **根拠**: コードの具体的な箇所を引用した根拠
- **補足**: 追加で見るべき観点
```

ルール:

1. 「同意する」「問題ない」だけで終わらせない
2. 必ずコード上の根拠を示す
3. 正しい指摘でも、補足すべき条件や影響がないか確認する
4. 反論する場合は、何が安全性や正しさを保証しているか明示する

## 統合レポート

複数レビューを統合できる場合は、以下に分類する。

| 条件 | 分類 | 信頼度 |
|---|---|---|
| 複数レビュアーが同じ問題を指摘し、根拠付きで同意 | Agreed | 高 |
| 片方のみ指摘し、相手が根拠付きで同意 | Agreed | 高 |
| 片方のみ指摘し、相手が根拠付きで否定または未確認 | Single | 中 |
| 解釈や前提が矛盾する | Disputed | 要判断 |

```markdown
## Code Review Report

### Summary
- Agreed: X件
- Single: Y件
- Disputed: Z件

### Agreed
#### [Critical/Warning/Info] 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **指摘**: 内容
- **根拠**: 複数レビューで一致した根拠
- **提案**: 修正案

### Single
#### [Critical/Warning/Info] 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **指摘者**: レビュアー名
- **指摘**: 内容と根拠
- **相手の見解**: 反論、未確認、または補足

### Disputed
#### 指摘タイトル
- **ファイル**: `path/to/file.ts:行番号`
- **見解A**: 内容と根拠
- **見解B**: 内容と根拠
- **人間に判断してほしい点**: 判断条件

### Review Process
- レビュー対象: diff / file / PR / symbol
- 使用したレビュアーまたはモデル
- 実行できなかった検証
```

## 制約

- 対象外ファイルを勝手に変更しない
- 推測ではなく実際のコードに基づく
- テスト未実行なら未実行と明記する
- 指摘がない場合も、残るリスクや確認範囲を短く書く
